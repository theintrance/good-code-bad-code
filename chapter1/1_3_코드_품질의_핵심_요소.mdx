---
title: '코드는 어떻게 소프트웨어가 되는가'
---

- 코딩 업무에 대해서가 아닌 근본적 성취에 대해서 이야기 했기에, 지금부터는 코딩 업무에 대한 조언을 제공한다.

    1. 코드는 읽기 쉬워야 한다.
    2. 코드는 예측 가능해야 한다.
    3. 코드를 오용하기 어렵게 만들라.
    4. 코드를 모듈화하라.
    5. 코드를 재사용 가능하고 일반화할 수 있게 작성하라.
    6. 테스트가 용이한 코드를 작성하고, 제대로 테스트하라.

### 1.3.1 코드는 읽기 쉬워야 한다.

```text
그릇을 준비한다. 이것을 A라고 부르자. ...
```

- 이 긴 브라우니 레시피에 대한 질문이 있다면 답을 찾을 수는 있겠지만 쉽지는 않다.
- 코드 또한 작성 후 다른 개발자가 코드를 읽고 이해해야 하는 상황이 반드시 온다.
- 가독성이 낮다면, 기능에 대한 잘못된 이해와 중요한 점을 놓칠 가능성이 생긴다.
- 코드에 대한 이해가 낮은 상태에서 전체 소프트웨어가 제대로 작동하는 것은 불가능에 가깝다.

### 1.3.2 코드는 예측 가능해야 한다.

- 코드에서는 “변수명, 데이터 타입, 일반적인 관행” 같은 단서로 무슨 일을 하는지와 무엇을 반환하는지에 대한  정신 모델을 구축한다.
- 만약 정신 모델과 어긋나는 어떤 일이 코드에서 일어나면, 아무도 모르게 버그가 코드 내로 유입되는 일이 많이 발생하게 된다.
- 잠시 동안은 문제가 없는 것처럼 보일 순 있으나, 점점 더 심각한 상황을 초래할 수 있다.
- 그렇기에, 코드가 예상을 벗어나는 일을 수행하지는 않는지 주의 깊게 살펴야 하고, 되도록 그런 코드를 작성하지 않도록 노력해야 한다.

### 1.3.3 코드를 오용하기 어렵게 만들라.

- 코드를 오용하기 쉽게 만들어 다른 개발자가 오용한다면 중요한 일이 수행되지 않거나, 눈에 띄지 않아 더 큰 문제를 야기할 수 있다.
- 코드를 오용하기 어렵거나, 불가능하게 만들면 코드를 그저 “작동”할 뿐 아니라 “잘 작동”할 가능성을 극대화 시킬 수 있다.

### 1.3.4 코드를 모듈화하라.

- 인터페이스를 잘 정의하여, 서로 다른 구성 요소 간 상호작용하는 지점을 최소화하라.
- 변화하는 요구 사항에 더 쉽고, 적은 코드를 작성하는데 도움이 된다.
- 또한 문서화에도 좋고, 코드에 대한 오해 가능성도 줄어든다.

### 1.3.5 코드를 재사용 가능하고 일반화할 수 있게 작성하라.

- 재사용성 - 어떤 문제를 해결하기 위한 무언가가 여러 가지 다른 상황에서도 사용될 수 있음을 의미
    - 핸드 드릴이 바닥 or 벽을 뚫는 행위
- 일반화성 - 서로 미묘하게 다른 문제들을 해결할 수 있음을 의미
    - 핸드 드릴이 뚫는 것 뿐 아니라, 나사를 돌릴 수도 있는 것
- 우리는 코드를 많이 작성함에 대한 대가를 받는 것이 아닌, 문제를 해결함에 대한 대가를 받는 것이기에 코드 라인이 적으면 적을 수록 좋다.
    - 많은 코드는 의도치 않게 다른 문제를 일으킬 수 있기 때문이다.
- 재사용할 수 있고 일반화되어 있다면 하나의 코드로 여러 문제를 해결 할 수 있다.

### 1.3.6 테스트가 용이한 코드를 작성하고 제대로 테스트하라

- 테스트는 코드가 동작하고, 멈추지 않고 계속 잘 실행하도록 보장하기 위해 필수적인 부분이다.
- 소프트웨어 시스템과 코드베이스는 너무 크고 복잡해 한 사람이 모든 세부 사항을 알 수 없고, 사람은 실수를 하는 존재다.
- 테스트와 테스트 용이성은 관련은 있지만, 고려하는 사항이 다르다.
- 테스트 - 코드 혹은 소프트웨어 전체를 테스트하는 것
    - 단위 테스트 - 일반적으로 개별 함수나 클래스와 같은 작은 단위의 코드를 테스트, 개발자가 가장 자주 작업하는 수준의 테스트
    - 통합 테스트 - 시스템을 구성하는 여러 구성 요소, 모듈, 하위 시스템을 연결하는 과정을 테스트
    - 종단간 테스트 - 처음부터 끝까지 전체 소프트웨어 시스템에서 작동의 흐름을 테스트
- 테스트 용이성 - “실제”코드를 가르키며 해당 코드가 얼마나 테스트하기를 적합한지를 나타냄
    - 작성 중인 코드의 테스트 용이성을 확인하기 위해 코드를 작성하면서 “어떻게 테스트할 것인가?”를 계속 자문하는 것이 좋다.
