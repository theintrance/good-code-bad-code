title: '코드 품질의 목표'
---

### 1.2.1 코드는 작동해야 한다

* "작동" 한다는 것에 여러가지 의미가 있을 수 있음
    * 코드가 예상한 대로 작동하는가?
    * 코드에 버그가 없는가?
    * CPU 점유율은 어떤가?
    * 보안 상 결함은 없는가?

* 제품에 따라 "작동" 한다는 것의 의미가 달라질 수 있음

### 1.2.2 코드는 작동이 멈추면 안된다.

* 코드는 한번 짜놨다고 해서 계속 잘 동작할거라는 보장이 없음
* 계속 작동시키기 위해 여러가지를 고려해야 함

1. 의존하는 다른 코드의 변경이 있을 수 있다.
2. 새로운 기능이 필요할 때 코드는 수정된다.
3. 시간이 지남에 따라 코드의 요구사항은 달라진다.

* 요구사항의 변화 등 여러 변화에 의해 코드가 중단되는 것을 막아야 한다.

### 1.2.3 코드는 변경된 요구사항에 적응할 수 있어야 한다.

* 그럼 어떤식으로 개발해야할까?

1. **방법 A**: 요구사항이 어떻게 바뀔지 정확히 예측하고 모든 잠재적 변화에 대응할 수 있는 코드를 몇 주 몇 달에 걸쳐 설계하고 모든 결정을 심사숙고하여 결정한다.

2. **방법 B**: 최대한 빨리 코드를 동작시키는 것을 목표로 요구사항의 변화를 전혀 고려하지 않고 무조건 빨리 작성한다.

* 방법 A는 너무 많은 시간이 걸리고 비용이 많이 든다.
* 방법 B는 코드가 변경되는 것을 고려하지 않았기 때문에 코드가 변경되면 작동하지 않을 수 있다.

* 두 극단적인 예시 그 사이 어딘가 정답이 있겠지만 어디인지는 아무도 알 수 없다.
* 이 Trade-off 를 잘 고려하여 조직 문화를 잘 구성해야 한다.

### 1.2.4 코드는 이미 존재하는 기능을 중복 구현해서는 안 된다.

* 같은 기능을 하는 코드를 여러번 쓰는 것을 중복 구현이라고 한다.
    * **예시 1**: 두 개의 서로 다른 클래스에서 같은 기능을 하는 코드를 작성하는 경우
    * **예시 2**: JPEG 이미지를 로드하는 기능을 직접 구현하는 경우

* 예시 1번의 경우 한 곳에서만 필요한 기능을 구현허고 필요한 쪽에서 참조하는 방식으로 구현할 수 있다.
* 예시 2번의 경우 라이브러리를 사용하는 방식으로 구현할 수 있다.
    * 단, 제품 개발의 경우 License 등의 문제로 라이브러리를 사용하지 못할 수도 있다.

* 중복 구현을 피할떄의 장점
    1. 코드 재사용성 증가
    2. 버그 가능성 감소
        * 이미 검증된 기능을 재사용하기 때문에 버그가 발생할 가능성이 적다.
    3. 유지보수 비용 감소
        * 라이브러리 사용 시 라이브러리를 업데이트 하므로써 새로운 기능을 사용할 수 있고 보안 상 이슈를 해결할 수 있다.
    4. 가독성 증가
        * 표준화된 라이브러리를 사용하므로써 코드를 이해하기 쉬워진다.
